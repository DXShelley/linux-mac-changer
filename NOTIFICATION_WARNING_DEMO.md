# 通知功能警告提示演示

## 场景：通知功能不可用时的用户提示

当用户配置的通知方式不可用时，脚本会显示以下警告并询问是否继续：

---

```
========================================
⚠️  通知功能不可用
========================================

当前配置的通知方式: url
问题:
  - 未配置 REMOTE_NOTIFY_URL
  - 或缺少 HTTP 客户端 (curl/wget)

⚠️  注意事项:

  如果您是通过 远程 SSH 连接到此主机:
    - 修改 MAC 后 IP 地址可能会改变
    - 您需要使用其他方式找到主机的 IP 地址
    - 可用方法:
      • 路由器 DHCP 客户端列表
      • 扫描局域网: sudo ./linux-mac-changer.sh scan 192.168.70.0/24
      • 直接连显示器查看 IP

  如果您是在 本地控制 此主机:
    - 无需担心，修改后可直接在终端看到新 IP
    - 本地文件通知将保存到: /tmp/new_ip.txt

💡 建议: 可以使用 'localfile' 模式作为保底方案

========================================

是否继续？(y/N):
```

## 用户选择

### 选择 1: 取消 (输入 N 或直接回车)

```
已取消操作
如需继续，请配置通知功能后重试:

方法1: 使用本地文件通知
  编辑脚本，设置: NOTIFY_METHOD="localfile"

方法2: 配置 URL 通知
  1. 启动通知服务器: python3 notification-server.py
  2. 编辑脚本，设置: REMOTE_NOTIFY_URL="http://YOUR_IP:8089"
```

### 选择 2: 继续 (输入 y)

```
✓ 继续执行（通知功能已禁用）
  MAC 修改功能将正常工作
  已自动切换到 localfile 通知模式
```

## 设计理念

1. **用户友好**: 清晰地说明问题和风险
2. **场景区分**: 区分远程SSH和本地控制两种场景
3. **提供方案**: 给出具体的解决方案和建议
4. **尊重选择**: 让用户决定是否继续
5. **自动降级**: 继续时自动切换到安全的localfile模式

## 检测逻辑

脚本会检测以下情况：

- `NOTIFY_METHOD="url"` 但未配置 `REMOTE_NOTIFY_URL`
- `NOTIFY_METHOD="url"` 但缺少 `curl` 或 `wget`
- `NOTIFY_METHOD="telegram"` 但未配置 `TELEGRAM_BOT_TOKEN` 或 `TELEGRAM_CHAT_ID`
- `NOTIFY_METHOD="telegram"` 但缺少 `curl`
- `NOTIFY_METHOD="all"` 且以上任一条件不满足

## 自动降级策略

如果用户选择继续，脚本会自动：
1. 将 `NOTIFY_METHOD` 设置为 `"localfile"`
2. 确保至少本地文件通知可用
3. MAC修改功能正常工作

## 安全保证

✅ **MAC修改功能完全独立于通知功能**
- 通知不可用不会阻止MAC修改
- 用户有明确的选择权
- 提供清晰的后果说明
- 自动降级到安全模式
